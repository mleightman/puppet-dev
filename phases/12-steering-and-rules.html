Phase 12 — Steering & Rules

Control Claude precisely with phrases and hard rules.

Claude wants to be helpful. Sometimes too helpful — it'll add features you didn't ask for, refactor code that was fine, or build the wrong thing because it guessed instead of asked. This phase teaches you how to keep Claude on track.

Steering Phrases

Short phrases that override Claude's instinct to do more than you asked:

Say ThisClaude Does This
"Do not implement yet"Reviews scope, rates confidence, identifies risks. No code.
"Just fix that one issue"Zero scope expansion. Fixes exactly what you pointed at.
"Continue to ask questions"Batches ALL questions before writing any code.
"I was just calling out one error"Fixes only the specific thing mentioned. Does not rewrite surrounding work.
"Interview style to talk through this plan"Asks you questions one at a time to collaboratively build a plan.
"Make sure you understand"Pauses and confirms understanding before proceeding. Prevents guessing.

These phrases go in your global CLAUDE.md so Claude knows them every session. You don't have to explain them each time — Claude reads the definitions from your protocol.

Hard Rules

Non-negotiable rules that Claude must follow. These also go in your CLAUDE.md:

RuleWhy It Exists
Nothing public without explicit approvalClaude might create public gists, repos, or deployments. Default to private.
No personal contact info in reposPhone numbers, emails get committed and are nearly impossible to fully remove.
No scope expansion without permissionFix one thing = fix one thing. Don't refactor the neighborhood.
No time estimatesAI time estimates are fiction. Don't let Claude guess.
Ambiguous input = ask, don't actOne clarifying question costs nothing. Unauthorized changes cost hours.
Never "find and fix bugs" broadlyClaude will invent bugs that don't exist and "fix" them, breaking working code. Always describe specific symptoms.
Never perform irreversible actions without verifyingDon't delete data before confirming the backup exists.

Workflow Rules

  • Batch work into small, deployable chunks. Never build everything in one session.
  • One code path per feature. No duplicate implementations.
  • Questions before implementing. Claude should ask questions in rounds of 3-6 before writing code.
  • Clean before documenting. Audit code before writing docs about it — otherwise docs canonize the bugs.
  • Review all doc edits twice before committing.
  • Before UI-heavy work, gather 2-4 visual references. Claude designs from references, not imagination.
  • Treat unscoped additions as scope creep. Claude will try to add adjacent features. Reject or defer them.

Bug Handling Protocol

When something breaks, follow this process:

  1. Describe the symptom specifically: "When I click the login button, I see a blank page instead of the form"
  2. Don't guess the cause: Let Claude trace the full code path
  3. Don't say "find and fix all bugs" — Claude will invent problems
  4. Check for duplicate implementations — sometimes Claude builds two versions of the same feature
  5. After fixing, trace the full lifecycle: create → use → cleanup → edge case

Edit Discipline

After every edit Claude makes, it reads the file back twice before continuing. This catches mistakes that "obvious" edits introduce. It's a rule in your CLAUDE.md — Claude does it automatically.

Why is this necessary?

Claude sometimes makes small mistakes when editing files — a missing bracket, a wrong variable name, an accidental deletion. Reading the file back twice forces Claude to verify what it actually wrote matches what it intended. It's like proofreading an email before sending it. The two reads ARE the forcing function.

Don't Worry About

  • Memorizing all steering phrases — they're in your CLAUDE.md, Claude reads them
  • Being "too strict" with rules — constraints make Claude better, not worse
  • Writing the perfect protocol on day one — add rules as you discover what goes wrong
