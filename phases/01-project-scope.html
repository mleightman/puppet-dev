Phase 1 — Project Scope

Write your project scope in web chat before touching any code.

What you'll learn

  • Write a project scope document that clearly defines what to build
  • Create an exclusion list that prevents scope creep
  • Understand acceptance criteria — how you'll know it's done
  • Use claude.ai web chat to brainstorm and narrow your idea

Before you build anything, you need to know what you're building. This phase happens in claude.ai (the briefing room), not in the terminal.

What You'll Do

  1. Open claude.ai in your browser
  2. Describe your app idea using the prompt below
  3. Let Claude ask you questions to narrow the scope
  4. End up with a project scope document — a clear brief for what you're building
  5. Save it somewhere you can find it later (Google Doc, text file, or Notes app)

The Prompt to Copy-Paste

Copy this prompt and paste it into claude.ai. Replace the bracketed part with your idea:

I want to build [describe your app idea in 1-2 sentences].

Interview me about it — ask me questions one at a time to help me narrow the scope. After we've talked through it, write me a project scope document with these sections:

1. What the app does (2-3 sentences)
2. Who it's for
3. Core features (3-5 max)
4. Exclusion list — what we are NOT building
5. Tech stack recommendation
6. Acceptance criteria — how I'll know it's done

Don't have an app idea yet? Start with "a simple to-do app" — you can always change direction later.

What Goes in a Scope Doc

Your scope doc is a contract between you and Claude. It says exactly what you're building and — just as importantly — what you're not building.

SectionWhat to Write
What the app does1-2 paragraphs describing the core idea
Who it's forWho will use this?
Core featuresThe 3-5 things the app must do
Exclusion listWhat is NOT in scope — features you're deliberately skipping
Tech stackClaude suggests this. For this tutorial, default to JavaScript/Node.js.
Acceptance criteriaHow you'll know it's done — specific, testable checkmarks

Example: Scope Doc for a To-Do App

Here's what a real scope doc looks like. Yours doesn't need to be fancier than this.

PROJECT SCOPE: Simple To-Do App

What it does:
A web app where you can add tasks, check them off, and delete them.
Tasks persist between visits (they're saved, not lost on refresh).

Who it's for:
Just me, for now. Maybe friends later.

Core features:
- Add a task with a text input and button
- Check off a task (strikethrough + move to "done" section)
- Delete a task
- Tasks are saved (persist between page refreshes)

NOT in scope (don't build these):
- User accounts or login
- Due dates or priorities
- Mobile app (web only)
- Sharing lists with other people
- Categories or tags

Tech stack:
JavaScript, Node.js, Express, PostgreSQL.
(Claude suggested this based on the "tasks persist" requirement.)

Acceptance criteria:
- I can add a task and it appears on the page
- I can check off a task and it moves to "done"
- I can delete a task and it disappears
- When I refresh the page, my tasks are still there
- The app is live at a public URL
Why is the exclusion list so important?

Claude is eager to help. If you say "build me a to-do app," Claude might add user accounts, due dates, drag-and-drop sorting, email reminders, and a mobile version — all without asking. The exclusion list is how you say "stop, just build what I asked for." You'll reference it constantly.

What is a "tech stack"?

A tech stack is the set of programming languages and tools Claude will use to build your app. Think of it like choosing materials for a house — wood vs. brick vs. steel. You don't need to understand the difference yet. For this tutorial, we'll use JavaScript and Node.js because they work well with the tools in later phases (GitHub, Railway, Claude Code).

When you chat with Claude in Phase 1, it may suggest a tech stack. If it suggests something other than JavaScript/Node.js, you can say: "Let's use JavaScript and Node.js for this project."

What are "acceptance criteria"?

Acceptance criteria are specific, testable statements about what "done" looks like. Not "the app works" — that's vague. Instead: "I can add a task and it appears on the page." You test this by literally doing it. If it works, that criterion is met.

Tips for Your Conversation with Claude

  • Start vague, get specific. "I want to build a to-do app" is a fine opening. Claude will ask follow-up questions.
  • Let Claude interview you. Say: "Ask me questions about what I want." Claude is good at this.
  • Say no to extras. If Claude suggests features you don't want, say "Not in scope."
  • Keep it small. Your first app should be simple enough to build in one session. You can always add features later.

Where to Save Your Scope Doc

Save it anywhere you can copy-paste from later. In Phase 6, you'll paste this scope doc directly to Claude Code and say "build me the starter version."

  • A Google Doc
  • A text file on your Desktop
  • Your Notes app
  • Copy it from the Claude chat and paste it somewhere

This scope doc becomes the foundation of everything. In Phase 8, it evolves into your CLAUDE.md — the file that teaches Claude your project every single session.
