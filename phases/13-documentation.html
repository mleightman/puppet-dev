Phase 13 — Documentation

The documentation stack and self-improving protocol.

What you'll learn

  • Know the 5 documentation layers and when to create each one
  • Understand the prevention rules loop — how solved problems make the protocol smarter
  • Write scope docs with explicit exclusion lists
  • Maintain your protocol as a living document

Your project builds up layers of documentation over time. Not all are needed from day one — start with CLAUDE.md and add layers as the project grows. Each layer serves a different purpose.

The Documentation Stack

LayerWhat It IsWhen to Create
CLAUDE.mdProject overview, architecture, key decisions, known issuesBefore first session
Scope docWhat to build, what NOT to build, acceptance criteriaBefore first build
Plan docsPre-build architecture for non-trivial workBefore each feature
Context docs (NCD)Session snapshots — what's working, broken, nextEvery 2-3 sessions
Solution docsPost-build: problem, root cause, fix, prevention rulesAfter every non-trivial fix

How the Layers Connect

  1. You write a scope doc in Phase 1 — it defines what to build
  2. The scope doc becomes the seed of your CLAUDE.md in Phase 8
  3. Before each feature, you write a plan doc (or Claude writes one during NUTS step 1)
  4. Every 2-3 sessions, you create a context doc (-ncd) — a snapshot of where things stand
  5. After every non-trivial fix, you write a solution doc — what broke, why, how you fixed it, and how to prevent it

The Prevention Rules Loop

This is how the protocol gets smarter over time:

  1. You solve a problem
  2. You document it in a solution doc, including Prevention Rules — rules that prevent the entire class of problem from recurring
  3. Those Prevention Rules feed back into your CLAUDE.md
  4. Next time Claude starts a session, it reads those rules and avoids the same mistake

Example: You discover that Claude keeps committing .env files to GitHub. You write a solution doc: "Problem: secrets committed to repo. Fix: added .env to .gitignore. Prevention Rule: always check .gitignore includes .env before first commit." That rule goes into your CLAUDE.md. Claude now checks .gitignore automatically.

What is a solution doc?

A solution doc records a problem you solved. Format:

  • Problem: What went wrong
  • Root Cause: Why it went wrong
  • Fix Applied: What you did to fix it
  • Prevention Rules: Rules that prevent this class of problem from happening again

You don't write these for every tiny fix — just for problems that taught you something worth remembering.

Protocol Maintenance

Your CLAUDE.md is a living document. It updates via conversation:

Type of ChangeWhere to Update
New commands, workflow rules, operating rulesGlobal ~/.claude/CLAUDE.md
Architecture, deployment, known issuesProject CLAUDE.md
Universal prevention rulesGlobal CLAUDE.md
Project-specific prevention rulesProject CLAUDE.md
Cross-session context, feedbackMemory files

The protocol gets better the more you use it. Every solved problem generates durable rules. Over time, your CLAUDE.md becomes smarter because it encodes every lesson learned.

Scope Docs and Exclusion Lists

Scope docs should always include an explicit exclusion list — what is NOT in scope. When Claude tries to add adjacent features (and it will), point at the list. This is your most powerful defense against scope creep.

Don't Worry About

  • Having all 5 layers on day one — start with CLAUDE.md, add the rest as you need them
  • Perfect documentation — good enough is better than nothing
  • Writing solution docs for every tiny fix — only for lessons worth remembering

Try It Now

Run this command in Claude Code to generate a context document for your project:

-ncd

Claude will read your entire codebase and produce a snapshot of what's working, what's broken, and what's next. This becomes your project's memory between sessions.
