Phase 4 — Environment Variables & Databases

Secrets, config, and where your app stores data.

You might not need this for your very first simple app. But read it now so you know the pattern — you'll use it as soon as your app needs to keep secrets safe or remember data between visits.

Environment Variables

Environment variables ("env vars") are secrets your app reads when it starts up — passwords, API keys, database addresses, anything you don't want visible in your code.

Why can't I just put passwords in my code?

Because your code lives on GitHub, and GitHub repos can be public. If you put a password in your code file and push it to GitHub, anyone can see it. Even in a private repo, it's bad practice — if the repo ever becomes public (accidentally or intentionally), your secrets are exposed.

Environment variables keep secrets separate from code. Your code says "read the password from the environment" — the actual password lives on the server (Railway), never in your GitHub repo.

How Env Vars Work

WhereWhat Happens
Your codeSays: "read the value of DATABASE_URL from the environment"
Railway dashboardStores the actual value: postgresql://user:pass@host/db
When your app startsRailway injects the value so your code can read it

How to Set Env Vars in Railway

  1. Open your project in the Railway dashboard
  2. Click on your service (the box that represents your app)
  3. Click the Variables tab
  4. Click New Variable
  5. Enter the name (like API_KEY) and the value (the actual secret)
  6. Click Add — Railway will automatically redeploy your app with the new variable

How to Tell Claude About Env Vars

When you're working with Claude Code, say things like:

  • "This API key should be an environment variable, not hardcoded"
  • "Read the database URL from process.env.DATABASE_URL"
  • "Make sure no secrets are in the code — use env vars"

Claude knows how to use environment variables. You just need to remind it when you see a secret in the code.

Databases

What is a database?

A database is a spreadsheet that your app reads and writes to automatically. When a user creates an account, adds a post, or saves a setting, that data goes into the database. When they come back later, the app reads the database to show their stuff.

Without a database, everything disappears when the page refreshes. With one, data persists.

When You Need a Database

  • User accounts or login
  • Saving data that persists between visits (posts, tasks, settings)
  • Anything where users create, edit, or delete content

If your app is just a static website with no user data, you don't need one.

How to Add a Database on Railway

  1. In your Railway project, click New
  2. Select Database
  3. Choose PostgreSQL (the most common choice)
  4. Railway creates the database and automatically generates a DATABASE_URL environment variable
  5. Tell Claude: "I have a PostgreSQL database on Railway. The connection URL is in the DATABASE_URL env var."
  6. Claude handles the rest — writing the code to connect, creating tables, querying data
What is PostgreSQL?

PostgreSQL (often called "Postgres") is a type of database — one of the most popular and reliable. It stores data in tables (like spreadsheets). There are other databases (MySQL, SQLite, MongoDB), but PostgreSQL is the standard choice for most web apps. Railway makes it easy to set up.

What is a "connection URL"?

A connection URL is an address that tells your app where the database is and how to log in. It looks like: postgresql://username:password@hostname:5432/database_name. You never need to type this — Railway generates it automatically. You just tell Claude it exists as an environment variable.

Do I need to learn SQL?

No. SQL is the language used to talk to databases. Claude writes all the SQL for you. You just describe what data you want to store and retrieve in plain English. Claude will write the correct SQL queries, create the tables, and set up the connections.

Don't Worry About

  • SQL syntax or database design
  • Database backups (Railway handles this)
  • Database performance or indexing
  • Why PostgreSQL over alternatives (it's the standard; trust the default)
