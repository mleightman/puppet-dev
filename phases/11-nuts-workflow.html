NUTS = No Uncritiqued Task Ships.

Without it, AI writes code fast, looks right, ships, breaks. NUTS catches problems at plan stage where fixing costs a paragraph edit instead of a debugging session.

What does NUTS stand for?

NUTS = No Uncritiqued Task Ships. The name captures the core principle: nothing goes live without being checked from multiple angles. It's not about perfection — it's about catching the obvious mistakes that rushing creates.

What is convergence?

Convergence means each critique pass finds fewer issues until there are zero actionable findings left. Pass 1 might find 6 problems. After fixing those, pass 2 finds 3. Pass 3 finds 1. Pass 4 finds 0. The work has been refined to the point where there's nothing left to improve. That's convergence.

The 8 Steps

StepWhat HappensPlain English
1. PlanResearch, identify gaps, write a planDescribe before touching code
2. Critique (-c)Adversarial passes until zero issuesFind everything wrong with the plan
3. Audit (pre-build)5-question diagnostic, Q1 = "what worries you?"Last cheap exit before code
4. Build (-build)Implement the critiqued planNow build it
5. Review (-c)Adversarial passes until zero issuesFind everything wrong with what was built
6. Audit (post-build)5-question diagnostic on implementationDid the code deliver the plan?
7. CompoundPush, then document the solutionWrite down what you learned
8. SummaryRecap + auto-archive completed TODOsThe final output

The Key Rules

  • Never skip to build. Plan → critique → audit → build. Always.
  • Critique until convergence. Not a fixed number of passes. Keep going until zero actionable findings.
  • Sequential passes: find issues → fix them → next pass. Not: find, find, find, then fix all at once.
  • Audit Q1 is always "what worries you?" — this is the gut-check. Honest answers only. No cheerleading.
  • Post-build audit is scoped to implementation. If a finding requires changing the architecture, flag it for the user — don't just change it.
  • Never skip compound. The knowledge compounds. Each documented solution makes future work faster.
  • Bug fixes follow the same cycle.

When to Scale Down

Full NUTS is for non-trivial work — multiple files, new architecture, judgment calls. For a typo fix or a config change, a quick -c pass is enough. Use your judgment:

Task SizeWhat to Do
Typo fix, config changeJust do it, maybe one -c pass
Small feature, bug fix-c → -i loop (Phase 10)
New feature, refactor, multi-file changeFull NUTS

How It Looks in Practice

Here's a real example of what NUTS feels like:

  1. You: "I want to add dark mode to my app. -nuts"
  2. Plan: Claude researches your codebase, writes a plan for how to add dark mode
  3. Critique: Claude runs -c on its own plan, finds 4 issues, fixes them, runs again, finds 1, fixes it, runs again, finds 0. Converged.
  4. Audit: 5 questions. Q1: "What worries me: the CSS specificity might clash with existing styles." Addresses it.
  5. Build: Claude implements the dark mode. Tests it.
  6. Review: -c on the built code. Finds a contrast issue. Fixes it. Next pass: clean.
  7. Audit: "Did the code deliver the plan?" Yes, all criteria met.
  8. Compound: Commits, pushes, documents what was done.
  9. Summary: "Added dark mode with CSS custom properties. Toggle persists in localStorage. 3 critique passes, 2 review passes. All converged."

The whole thing might take 10-20 minutes. The result: dark mode that actually works, is documented, and was checked from multiple angles before shipping.
