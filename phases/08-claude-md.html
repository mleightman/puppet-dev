Phase 8 — CLAUDE.md

Teach Claude your project — the file it reads every session.

CLAUDE.md is the most important file in your project. Claude reads it at the start of every session. It's how Claude knows what your project is, how you work, and what rules to follow. Without it, every session starts from zero. With it, Claude picks up right where you left off.

The Two-Tier System

There are two CLAUDE.md files, and they work together:

FileLocationWhat It ContainsApplies To
Global CLAUDE.md ~/.claude/CLAUDE.md Your full operating protocol — commands, workflow rules, steering phrases, NUTS flow Every project, every session
Project CLAUDE.md CLAUDE.md in your project root Project-specific info — tech stack, architecture, deployment, known issues Only this project

When both exist, the project-level file takes precedence for that project. Think of global as "how I work" and project as "what I'm working on."

Start from Your Scope Doc

Remember the scope doc from Phase 1? That's the seed of your project CLAUDE.md. Tell Claude:

Read my code and write me a CLAUDE.md for this project. Here's the scope doc I started with:

[paste your scope doc from Phase 1]

Claude will expand it into a proper CLAUDE.md with your tech stack, file structure, commands, and project context.

What Goes in the Global CLAUDE.md

The global file is your operating protocol — it applies to every project. It contains:

SectionWhat It Does
Session greetingA message Claude displays at the start of every session to confirm the protocol loaded
About the UserWho you are, your experience level, how you prefer explanations
CommandsYour custom command set (-c, -build, -status, etc.)
Steering phrasesShort phrases that control Claude's behavior ("Do not implement yet")
Workflow rulesHow to approach work (batch small, scope first, critique before build)
Operating rulesHard rules Claude must follow (nothing public without approval, etc.)

You'll build this up over time. Start simple and add rules as you learn what works.

The Session Greeting

Add a startup message so you know Claude loaded the protocol:

**On first message of every session**, display:
`Puppet Developer Installed. Built by Puppet Dev Studios.`

If you see this greeting, Claude read your instructions. If you don't, something didn't load — restart the session.

The "About the User" Section

This shapes every interaction. A senior engineer gets technical explanations. A first-time coder gets plain language. Tell Claude who you are:

## About the User

Matt. Beginner coder. Explain technical concepts in plain language.
Don't over-explain things I already understand from context.

What Goes in the Project CLAUDE.md

The project file is specific to one app:

SectionWhat It Does
About the ProjectWhat the app does (from your scope doc)
Tech StackLanguages, frameworks, database, hosting
StructureKey files and folders, what each one does
CommandsHow to run, test, and deploy the app
Known IssuesBugs, workarounds, things to watch out for

Memory System

Beyond CLAUDE.md, Claude has a memory system for cross-session context — things that would take more than 2 minutes to rediscover.

SystemWhat It DoesHow to Use It
MEMORY.md A curated index of memories stored in ~/.claude/projects/.../memory/ "Remember that we decided to use webhooks for notifications"
claude-mem plugin Auto-captures observations, searchable across sessions "What do you remember about the payment flow?"

Memory is for things that don't belong in CLAUDE.md but you don't want to repeat every session — feedback you've given, recurring patterns, project phase.

Git Workflow

Now that you have CLAUDE.md, here's the actual shipping workflow Claude handles for you:

  1. You make changes on a feature branch
  2. Claude creates a PR (pull request) via the gh CLI
  3. You review and merge the PR
  4. The code auto-deploys to Railway on push to main
What is a pull request (PR)?

A pull request is a proposal to merge changes from one branch into another. It's like saying "I've made these changes on my draft copy — here's what I changed, please review and merge it into the main version." Claude creates PRs for you using the gh CLI (GitHub's command-line tool). You just say "commit and push" or use the -cp command.

What is the gh CLI?

gh is GitHub's official command-line tool. It lets Claude create pull requests, check CI status, and merge code — all from the terminal. You don't need to learn it. Claude runs it. You just tell Claude what you want ("create a PR", "merge this") and Claude handles the gh commands.

Don't Worry About

  • Writing the perfect CLAUDE.md on day one — it grows and improves over time
  • The global vs. project distinction being confusing — start with just a project CLAUDE.md
  • Memory management — Claude handles the mechanics, you just say "remember this"
  • Git branching strategy — Claude handles branches and PRs
