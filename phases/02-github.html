Phase 2 — GitHub

Your code's home — create an account and your first repository.

GitHub is where your code lives. Think of it as Google Docs for code — it stores every version of every file, lets you undo mistakes, and lets multiple people work on the same project without stepping on each other.

Create Your Account

  1. Go to github.com
  2. Click Sign up
  3. Enter your email, create a password, choose a username
  4. Verify your email
  5. You now have a GitHub account

Create Your First Repository

A repository ("repo") is a folder for your project. Everything related to your app — code, images, config — lives inside it.

  1. Click the + icon in the top-right corner of GitHub
  2. Click New repository
  3. Name it something like my-first-app (lowercase, hyphens instead of spaces)
  4. Leave it set to Public (you can change this later)
  5. Check Add a README file
  6. Click Create repository

That's it. You now have a GitHub repo with your name on it.

Key Concepts

You'll hear these terms constantly. Here's what they mean:

TermWhat It IsAnalogy
Repository (repo) A project folder that tracks every change ever made to every file inside it A Google Drive folder with infinite version history
Commit A snapshot of your project at one moment in time. Each commit has a message explaining what changed and why. Pressing "Save" in a video game — you can always reload from that save point
Branch A separate copy of your project where you can make changes without affecting the main version. When you're happy with the changes, you merge the branch back. Making a "draft" copy of a document, editing it, then replacing the original when the draft is ready
Push Upload your local changes to GitHub so they're saved online Syncing your local files to the cloud
Pull Download the latest changes from GitHub to your computer Refreshing your local folder to get the latest cloud version
What is .gitignore?

.gitignore is a file that tells Git which files to skip — they won't be uploaded to GitHub. You use it for:

  • Secrets — passwords, API keys, anything private (because GitHub repos can be public)
  • Local config — files specific to your computer that won't work on someone else's
  • Junk files — things your OS or editor creates automatically (like .DS_Store on Mac)

Claude will set up your .gitignore for you. You just need to know it exists so you don't accidentally commit (upload) passwords to GitHub.

Public vs. Private repos?

Public: Anyone can see your code. Good for portfolios and open source. Free.

Private: Only you (and people you invite) can see it. Also free on GitHub.

For learning, public is fine. If your app handles real user data or has secrets, use private. You can always switch later.

Don't Worry About

These things exist but you don't need them yet:

  • GitHub Actions (automated workflows — Phase 14 territory)
  • Pull requests (you'll learn these in Phase 8)
  • Branching strategies (just use the main branch for now)
  • GitHub Issues, Projects, or Wikis
