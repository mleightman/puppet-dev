Phase 10 — The -c -i Loop

The core critique-and-implement pattern that powers everything.

What you'll learn

  • Use the -c (critique) and -i (implement) commands to refine any piece of work
  • Understand convergence — when critique finds nothing left to fix
  • Know when to use the loop standalone vs inside the full NUTS workflow

Before learning the full NUTS workflow, learn the one interaction pattern that makes everything else work. This is the single most important thing in this entire guide.

The Loop

  1. Claude does something (writes a plan, builds code, makes a suggestion)
  2. You type -c — Claude critiques its own work adversarially, finding everything wrong
  3. You type -i — Claude implements all valid fixes from the critique, no questions asked
  4. Repeat until clean

That's it. -c then -i, over and over, until there's nothing left to fix.

Why This Works

Claude is good at building things. Claude is also good at finding problems with things. But it doesn't do both at the same time. When you ask Claude to critique its own work, it switches from "builder" mode to "reviewer" mode — and it's surprisingly honest about what's wrong.

The loop forces Claude to check its own work before you ship it. Think of it as a built-in quality check that costs you two keystrokes.

When to Use It

Before NUTSRough out an idea with the loop, then formalize into a NUTS plan
After NUTSQuick polish pass on something that already went through the full flow
Instead of NUTSFor small tasks that don't warrant the full 8-step cycle

When the total hits zero (or only low/informational items remain), it's converged. Ship it.

What does "converged" mean?

Convergence means the critique passes are finding fewer and fewer issues until there's nothing actionable left. Pass 1 might find 6 issues. After you fix those (-i), pass 2 finds 4. After fixing those, pass 3 finds 2. Pass 4 finds 0. That's convergence — the work has been refined to the point where there's nothing left to improve.

Why does Claude critique its own work?

When you ask Claude to build something, it's in "builder" mode — focused on creating. When you then ask it to critique, it switches to "reviewer" mode — focused on finding problems. These are different mindsets. The loop forces Claude to use both, which catches mistakes that the builder mode missed.

Try It Now

Open Claude Code and try this:

  1. Ask Claude to do something: Write me a simple landing page for a to-do app
  2. Type -c
  3. Read the critique — see what Claude found
  4. Type -i
  5. Type -c again — see if there are fewer issues
  6. Repeat until converged

You'll see the pattern immediately: each pass finds less, until the work is clean. This is the foundation of everything in Phase 11 — The NUTS Workflow. For the full set of commands beyond -c and -i, see Phase 12 — Steering & Rules.
