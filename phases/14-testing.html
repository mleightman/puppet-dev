Phase 14 — Testing

Let Claude write tests — you just need to know they exist.

What you'll learn

  • Ask Claude to write and run tests for your features
  • Understand what "tests pass" means and when to ship
  • Know where testing fits in the NUTS workflow
  • Request specific tests (not "test everything")

You don't need to understand how to write tests. You just need to know they exist, that Claude writes them for you, and when to ask for them.

What Tests Are

What is a test?

A test is a small program that checks if your main program works correctly. Instead of you manually clicking every button to verify things work, a test does it automatically in seconds. Claude writes these for you — you just need to ask and then check if they pass (green) or fail (red).

Tests are code that checks if your other code works correctly. Think of it as an automatic checklist — instead of you manually clicking every button to verify things work, tests do it for you in seconds.

How to Use Tests with Claude

  1. After Claude builds a feature, say: Write tests for what you just built
  2. Claude creates test files that verify the feature works
  3. Run the tests: npm test (or whatever your project uses)
  4. If tests pass: the code works as intended. Green light to ship.
  5. If tests fail: Claude reads the failures and fixes the code

When to Ask for Tests

SituationWhat to Say
After building a significant feature"Write tests for the feature you just built"
Before deploying to production"Run the tests and make sure everything passes"
Something that worked stops working"Write a test that proves this feature works, then run it"
You want to be specific"Write tests for the login flow" (not "write tests for everything")

Key Commands

CommandWhat It Does
npm testRun all tests
npm run test:single -- path/to/test.jsRun one specific test file

You can also just tell Claude run the tests and it will execute them, read the results, and tell you what passed or failed. (If you need a refresher on running commands in the terminal, see Phase 6 — Claude Code CLI.)

The NUTS Connection

Tests fit into the NUTS workflow at two points:

  • Build step (step 4): Claude tests each component as it builds
  • Review step (step 5): Tests help verify nothing is broken

Tests are part of the quality system, not separate from it.

Testing Tips

  • Be specific. "Write tests for the login flow" is better than "write tests for everything" — too broad causes Claude to write bloated, slow tests
  • If Claude says "all tests pass," ship it. That's the green light.
  • Tests catch bugs before your users do. A failing test is better than a user reporting a broken feature.
  • Don't stress about test coverage. Some tests are better than no tests. Test the important flows first.

Don't Worry About

  • Understanding test syntax — Claude writes and reads them
  • Test frameworks or configuration — Claude sets those up
  • 100% test coverage — focus on the features that matter most
  • Writing tests yourself — that's Claude's job
